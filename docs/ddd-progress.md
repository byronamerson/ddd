## What This Project Is

The `ddd` package is a practice sandbox for developing and refining
a contract-first, test-driven workflow for LLM-collaborative R
development. It is not a production tool — it is the working
laboratory where the ideas in the Lingua principles, the R
Development Principles, and the Contract-First Development document
are tested against real coding practice.

The project lives at `ddd` on GitHub (private repo). It is an R
package, but it is also more than a package — the directory structure
accommodates three distinct modes of work.

---

## The Three-Gear Architecture

The central design decision of the project is the **three-gear
workflow**, documented in `docs/three-gear-workflow.md`. It addresses
a tension: contract-first development produces reliable code but
assumes you know what to build, while applied science work often
starts without that certainty.

### Gear 1 — Exploration (`exploration/`)

Rough scripts, plots, data investigation. Code is investigative and
disposable. The LLM behaves as a reactive helper — no contracts, no
tests, no structure suggestions. The sniff test (do the values make
physical sense?) is the only quality control that is always on.

Context: `exploration/CONTEXT.md`

```
exploration/
  scripts/       # rough .R scripts
  data/          # working data files
  figures/       # exploratory plots
  notes/         # scratch observations and questions
```

### Gear 2 — Collaborative Design (`design/`)

The functional goal exists but the specification is incomplete. The
LLM becomes a collaborative designer — asking questions, proposing
decomposition, surfacing edge cases. Code is prototype-quality.
Design decisions are recorded as they emerge. No formal tests yet.

Context: `design/CONTEXT.md`

```
design/
  prototypes/    # working scripts that prototype a function
  notes/         # design decisions, conversation summaries
```

### Gear 3 — Contract-First Development (package root)

Full discipline per `CLAUDE.md`: Roxygen first, tests derived from
contract, implementation last. The LLM is a constrained implementer
operating within the contract and tests.

Context: `CLAUDE.md`

```
R/               # package source
tests/testthat/  # testthat 3e tests
man/             # generated by roxygen2
```

### The gears cycle

The workflow is not a one-way escalator. Building a tested function
in Gear 3 often produces output that raises new questions, sending
you back to Gear 1. A design conversation may surface a data
question that requires exploration. Multiple loops between gears
before a function reaches the package are normal. Each cycle adds
understanding. A return to Gear 1 is a deepening, not a restart.

### The crystallisation principle

Code does not move mechanically between gears — understanding moves.
When a function graduates to the package, the contract is written
fresh, informed by what was learned in exploration and design. The
prototype is a reference, not a template. This prevents carrying
over the rough edges and implicit assumptions of exploratory code.

## What Has Been Built

### Package functions (Gear 3 — complete)

**`ddd_convert_discharge()`** — Converts discharge from cfs to cms.
The first function built using the full contract-first cycle,
done interactively. Roxygen written first, 16 tests derived from
the contract and confirmed RED, then implementation written to pass
them. A deliberately simple function chosen to exercise the workflow
without domain complexity getting in the way.

**`assess_record()`** — Evaluates a temperature time series record
for completeness and data quality. Contract extracted from the
temperheic project (an existing R package), implementation stripped.
A fresh Claude Code session with zero prior context derived 33 tests
from the contract alone, then implemented the function to pass all
33. The implementation closely matches the original temperheic
version — a strong validation that the contract carries sufficient
information for independent implementation.

All 49 tests pass. Package loads cleanly.

### Exploration work (Gear 1 — complete, ready for Gear 2)

**Run-length analysis** — An exploration of sequential runs above or
below a threshold in daily hydrographs, using USGS gage data from
Meacham Creek OR (snowmelt regime) and Elder Creek CA (rain-driven
regime). The exploration script (`exploration/scripts/run-length-explore.R`)
is working and covers data pull, season filtering, period assignment,
run detection via `rle()`, and both summary and detail output.

A Gear 2 handoff prompt has been written
(`design/notes/gear2-run-length-prompt.md`) with eight design
questions that surfaced during exploration: function boundary, input
contract, NA handling policy, season filtering scope, generality
beyond discharge, return structure, comparison operator interface,
and wrapping `rle()` index arithmetic.

## Key Insights from the Journey

### Tests as pre-registered expectations

The pivotal reframe from early sessions: tests before implementation
are not a tautology. They define what "correct" means before the
output exists. They are the sniff test written down in code — you
form expectations before seeing the result, so the result cannot
quietly reshape your expectations. Same principle as checking a
hydrograph against your mental model before trusting it.

### The contract carries the specification

The `assess_record()` experiment demonstrated that a well-written
Roxygen block carries enough information for an LLM with zero prior
context to derive tests and produce a correct implementation. This
validates the core claim of the contract-first workflow: the Roxygen
block is simultaneously a specification, a prompt, and a test
scaffold.

### The three-gear model resolves a real tension

Forcing contract-first discipline on exploratory work creates
friction without adding value. Leaving exploratory work unstructured
forever produces fragile scripts. The gears give each mode of work
its own space, its own context, and its own rules, with explicit
transition signals between them.

### Contract unification (in progress)

A Codex adaptation session surfaced an important conceptual
refinement: Roxygen contracts (packages) and structured contract
comments (scripts) are the same grammar in two rendering contexts.
Scripts should get a function-boundary contract by default (opt-out,
not opt-in). Moving a function from a script to a package means
re-rendering the contract as Roxygen, not rethinking it. The
guiding documents (`lingua.md`, `contract-first.md`) have not yet
been revised to reflect this unification.

### The contract threshold test

Not every function needs a full contract. The test: does the
function encode domain knowledge, a non-obvious decision, or a
verifiable constraint that cannot be fully recovered from the name
and signature alone? If yes, it needs a contract. If the name plus
the body together constitute the full specification, it does not.
In applied science code, most helpers encode domain knowledge, so
this is nearly as strict as "all of them" in practice — but it
does not mandate noise on mechanical wrappers.

## Open Threads

### Package development (Gear 3)

The `next-session-prompt.md` lists four options:

1. **Forward pass** — pick a new function, write the contract from
   domain knowledge, derive tests, implement. Tests muscle memory
   on new work rather than reverse-engineering.
2. **Implementation comparison** — diff the `assess_record()`
   implementation against the temperheic original.
3. **Discuss-first workflow** — start from natural language (Lingua
   §2), let discussion produce the contract, derive tests, implement.
   Exercises the full chain.
4. **Retrospective audit** — take a different temperheic function,
   extract the contract, derive tests, check if the existing
   implementation passes.

### Run-length analysis (Gear 2 ready)

The exploration is complete and the Gear 2 handoff prompt is written.
The next step is a design conversation to work through the eight
questions and produce a specification stable enough for Gear 3.

### Document revisions (cross-cutting)

`docs/lingua.md` and `docs/contract-first.md` need revision to
unify the contract model (scripts and packages share the same
grammar). `r-principles.md` may need a more directive statement
about contract requirements. These revisions were requested but
not yet delivered.
