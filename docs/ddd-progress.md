## What This Project Is

The `ddd` package is a practice sandbox for developing and refining
a contract-first, test-driven workflow for LLM-collaborative R
development. It is not a production tool — it is the working
laboratory where the ideas in the Lingua principles, the R
Development Principles, and the Contract-First Development document
are tested against real coding practice.

The project lives at `ddd` on GitHub (private repo). It is an R
package, but it is also more than a package — the directory structure
accommodates three distinct modes of work.

---

## The Three-Gear Architecture

The central design decision of the project is the **three-gear
workflow**, documented in `docs/three-gear-workflow.md`. It addresses
a tension: contract-first development produces reliable code but
assumes you know what to build, while applied science work often
starts without that certainty.

### Gear 1 — Exploration (`exploration/`)

Rough scripts, plots, data investigation. Code is investigative and
disposable. The LLM behaves as a reactive helper — no contracts, no
tests, no structure suggestions. The sniff test (do the values make
physical sense?) is the only quality control that is always on.

Context: `exploration/CONTEXT.md`

```
exploration/
  scripts/       # rough .R scripts
  data/          # working data files
  figures/       # exploratory plots
  notes/         # scratch observations and questions
```

### Gear 2 — Collaborative Design (`design/`)

The functional goal exists but the specification is incomplete. The
LLM becomes a collaborative designer — asking questions, proposing
decomposition, surfacing edge cases. Code is prototype-quality.
Design decisions are recorded as they emerge. No formal tests yet.

Context: `design/CONTEXT.md`

```
design/
  prototypes/    # working scripts that prototype a function
  notes/         # design decisions, conversation summaries
  skills/        # reusable agent skills (test-from-contract, etc.)
  inter-rater/   # inter-rater reliability experiment infrastructure
```

### Gear 3 — Contract-First Development (package root)

Full discipline per `CLAUDE.md`: Roxygen first, tests derived from
contract, implementation last. The LLM is a constrained implementer
operating within the contract and tests.

Context: `CLAUDE.md`

```
R/               # package source
tests/testthat/  # testthat 3e tests
man/             # generated by roxygen2
```

### The gears cycle

The workflow is not a one-way escalator. Building a tested function
in Gear 3 often produces output that raises new questions, sending
you back to Gear 1. A design conversation may surface a data
question that requires exploration. Multiple loops between gears
before a function reaches the package are normal. Each cycle adds
understanding. A return to Gear 1 is a deepening, not a restart.

### The crystallisation principle

Code does not move mechanically between gears — understanding moves.
When a function graduates to the package, the contract is written
fresh, informed by what was learned in exploration and design. The
prototype is a reference, not a template. This prevents carrying
over the rough edges and implicit assumptions of exploratory code.

### How the three-gear model emerged

The model was prompted by practical experience with Codex: the
contract-first AGENTS.md was being applied to exploratory work,
creating friction without adding value. The documents assumed you
always start from a contract, so there was no guidance for the
discovery phase. The critical insight was that Gear 2 (collaborative
design) is a distinct mode, not just a transition.

### Context isolation strategy

For agentic tools (Claude Code, Codex), the gears should be separate
directories with their own CLAUDE.md / AGENTS.md to prevent context
pollution. For interactive chat (Claude desktop app, claude.ai),
stating the gear at the start of the session is sufficient.

## What Has Been Built

### Package functions (Gear 3 — complete)

**`ddd_convert_discharge()`** — Converts discharge from cfs to cms.
The first function built using the full contract-first cycle,
done interactively. Roxygen written first, 16 tests derived from
the contract and confirmed RED, then implementation written to pass
them. A deliberately simple function chosen to exercise the workflow
without domain complexity getting in the way.

**`assess_record()`** — Evaluates a temperature time series record
for completeness and data quality. Contract extracted from the
temperheic project (an existing R package), implementation stripped.
A fresh Claude Code session with zero prior context derived 33 tests
from the contract alone, then implemented the function to pass all
33. The implementation closely matches the original temperheic
version — a strong validation that the contract carries sufficient
information for independent implementation.

All 49 tests pass. Package loads cleanly.

### `tally_runs()` — contract and tests complete, not yet implemented

**Roxygen contract** written for `tally_runs()` in `R/tally_runs.R`.
Tallies sequential runs above or below a threshold in a time series,
partitioned by water year, calendar year, or custom season. Seven
parameters, detailed `@return` structure, NA behaviour, period
partitioning rules, and input validation. The contract emerged from
Gear 1 exploration and Gear 2 design conversations.

**49 tests** derived from the contract by a bespoke prompt (zero
implementation context). All fail against the stub — RED phase
confirmed.

**A generalized test-from-contract skill** was built at
`design/skills/test-from-contract/` with methodology references.

### Inter-rater reliability experiment (complete)

Three independent Claude Code CLI agents ran the test-from-contract
skill against the same `tally_runs()` Roxygen contract. Each ran in
headless mode (`claude -p`) with `--disallowedTools` blocking access
to the existing test file and design notes. File access logs confirm
no contamination occurred.

Results (full analysis in `design/inter-rater/comparison-analysis.md`):

- **Test counts:** 40, 41, 43 across three runs (vs. 49 from bespoke)
- **33 test intents converged (3/3):** The high-confidence core.
  The contract clearly implies these tests.
- **~7 intents at majority (2/3) or singleton (1/3):** Divergence
  points to contract ambiguities or scope judgment.
- **No contradictory expectations:** Where agents disagreed, it was
  about coverage scope, not about what correct behaviour looks like.
- **2 contract gaps flagged universally:** "natural unit" inference
  and MMDD HHMM interaction with Date-class input.
- **3 assumptions converged universally:** duration = end − start,
  valid `duration_units` = difftime units, WY follows USGS convention.

Decision: **keep inter-rater testing for complex functions at
contract-finalization time.** Not for every function. The technique
is most valuable when the contract has multiple parameters, domain
rules, and non-obvious edge cases. For simpler functions, a single
derivation run suffices.

### Exploration work (Gear 1 — in progress)

**Run-length analysis** — Exploration of sequential runs above or
below a threshold in daily hydrographs, using USGS gage data from
Meacham Creek OR and Elder Creek CA. The exploration script has a
known bug in period assignment (scalar `period_type` in vectorized
`if_else()`). Design decisions settled during exploration are
documented in `design/notes/run-length-design-decisions.md`.

## Key Insights from the Journey

### Tests as pre-registered expectations

The pivotal reframe from early sessions: tests before implementation
are not a tautology. They define what "correct" means before the
output exists. They are the sniff test written down in code — you
form expectations before seeing the result, so the result cannot
quietly reshape your expectations. Same principle as checking a
hydrograph against your mental model before trusting it.

### The contract carries the specification

The `assess_record()` experiment demonstrated that a well-written
Roxygen block carries enough information for an LLM with zero prior
context to derive tests and produce a correct implementation. This
validates the core claim of the contract-first workflow: the Roxygen
block is simultaneously a specification, a prompt, and a test
scaffold.

### The three-gear model resolves a real tension

Forcing contract-first discipline on exploratory work creates
friction without adding value. Leaving exploratory work unstructured
forever produces fragile scripts. The gears give each mode of work
its own space, its own context, and its own rules, with explicit
transition signals between them.

### Gear 2 is where the interesting work happens

Collaborative design — where the LLM asks questions, proposes
structure, and a specification emerges through dialogue — is not
merely a transition between exploration and contract-first
development. It is a distinct mode of work where most applied
science specifications actually get created.

### Context pollution is the practical enemy

When a single CLAUDE.md applies contract-first rules to all work in
a project, exploratory work becomes laborious. The fix is context
isolation: each gear gets its own context document.

### Contract unification (in progress)

Roxygen contracts (packages) and structured contract comments
(scripts) are the same grammar in two rendering contexts. Scripts
should get a function-boundary contract by default. Moving a
function from a script to a package means re-rendering the
contract as Roxygen, not rethinking it.

### The contract threshold test

Not every function needs a full contract. The test: does the
function encode domain knowledge, a non-obvious decision, or a
verifiable constraint that cannot be fully recovered from the name
and signature alone?

### Agentic inter-rater reliability works — with calibration

The experiment confirmed that independent test-derivation runs
produce meaningful divergence that reveals contract ambiguities.
About 80% of test intents converge across all runs (the contract is
specific enough), but the remaining 20% divergence points directly
to underspecified clauses and implicit assumptions that should be
made explicit. Two runs capture most of the value; three add
majority-vote resolution. The technique is worth deploying on
complex functions but overkill for simple ones.

### The workflow is more ceremony than daily applied work needs

The full contract-first cycle with test derivation is more
long-winded than typical applied science work. Most work lives in
Gear 1 and the applied analysis workflow — scripts, plots,
composing existing tools. The full machinery fires only when
something crosses the threshold into "this needs to be reliable
across cases and worth keeping." That is occasional by nature, and
when it happens, doing it right earns its keep.

## Open Threads

### `tally_runs()` implementation (Gear 3 — ready)

The contract has been validated by the inter-rater experiment. Five
contract revisions are recommended before implementation:

1. Specify the "natural unit" inference algorithm for `duration_units = NULL`
2. State that MMDD HHMM boundaries require POSIXct input
3. Clarify MMDD validation boundary (is Feb 30 accepted?)
4. Make duration = end − start explicit in the contract
5. State that "continuous" means consecutive rows, not time-adjacent

After contract tightening, proceed to implementation.

### Run-length exploration (Gear 1 → Gear 2)

The exploration script has a known bug. Once fixed, either continue
exploring (Elder Creek, different thresholds) or move to Gear 2 for
the design conversation.

### Document revisions (cross-cutting)

`docs/lingua.md` and `docs/contract-first.md` need revision to
unify the contract model (scripts and packages share the same
grammar). The contract-first doc has been updated with the
inter-rater experiment results (§3).
