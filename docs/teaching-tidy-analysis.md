# Teaching tidy analysis as three operations: add, summarize, expand

**Every tidy data analysis reduces to three moves: enrich rows by adding columns, collapse rows by summarizing groups, and expand rows by joining tables.** This framework gives analysts who already know `mutate()`, `summarize()`, and `left_join()` a higher-level mental model — a workflow philosophy rather than a function reference. The conceptual foundations trace directly to Hadley Wickham's tidy data principles and split-apply-combine strategy, but the three-operation framing synthesizes ideas scattered across the tidyverse ecosystem into a single teachable sequence. What follows is a research synthesis of the theoretical, pedagogical, and applied foundations for teaching this workflow to graduate students and junior analysts.

## Tidy structure makes row enrichment the natural first move

Wickham's 2014 "Tidy Data" paper established the constraint that makes everything else work: **each variable forms a column, each observation forms a row, each type of observational unit forms a table**. This structure, equivalent to Codd's third normal form restated for analysts, has a crucial consequence the paper states directly: "Tidy data is particularly well suited for vectorised programming languages like R, because the layout ensures that values of different variables from the same observation are always paired." When every column is a variable and every row is an observation, adding a new column via `mutate()` automatically pairs the new value with the correct observation. No index alignment, no row matching — the structure guarantees correctness.

This property transforms `mutate()` from a convenience function into a workflow philosophy. The tidy data paper itself demonstrates this with a tuberculosis dataset: start with tidy counts, add a population column (via join), then compute rates as a new column. Wickham notes that "in tidy form, adding variables for population and rate is easy — they are just additional columns." The implication is profound: **the path from raw data to analytical result is a sequence of column additions that progressively make each row self-contained for the target calculation**. The paper's discussion of denormalization — merging normalized tables back into one — frames joins as the mechanism for column additions that cross table boundaries.

R for Data Science (2nd edition) reinforces this by organizing dplyr verbs into four groups based on what they operate on: **rows** (filter, arrange), **columns** (mutate, select), **groups** (group_by, summarize), and **tables** (joins). This taxonomy implicitly teaches the workflow: first work within columns to enrich, then shift to groups to collapse, then bridge tables to expand. The deliberate placement of joins in Chapter 19 — after students master single-table operations — creates a pedagogical progression that mirrors the analytical workflow itself.

## Column accumulation as workflow philosophy, not just syntax

The most effective teaching of this pattern comes from sources that frame `mutate()` as progressive enrichment rather than isolated column creation. Jenny Bryan's STAT 545 course demonstrates this through what she calls "building columns sequentially": creating a temporary reference value via `mutate()`, then using it to compute a relative measure in the same pipeline. Her key pedagogical move is the immediate sanity check — "The Canadian values for `gdpPercapRel` better all be 1!" — teaching that **each enrichment step should be verified before the next**. Bryan's "Grand Finale" exercise chains `select → group_by → mutate (lag differences) → summarize → top_n → arrange`, showing the complete arc from enrichment through summarization.

Bryan's dedicated "row-oriented-workflows" repository teaches the deeper philosophy through negative examples. Exercise ex01, titled "Leave it in the data frame," demonstrates the "evil of creating copies of certain rows of certain variables, using Magic Numbers and cryptic names, just to save some typing." The core message: don't extract data to work with it separately; keep it in the data frame and add columns. Exercise ex05 makes the case for column-wise thinking explicitly: "building a tibble column-by-column is less aggravating than building rows, then row binding."

The clearest articulation of enrichment-as-preparation comes from the Ocean Health Index science training materials, which demonstrate `mutate(gdp = pop * gdpPercap)` followed by `mutate(max_gdp = max(gdp))`, with the commentary: "You are now set up nicely to maybe take a proportion." This framing — **mutate sets up rows for a downstream computation** — captures the workflow philosophy in a single sentence. The signal that you're doing this correctly is that each `mutate()` call brings every row closer to containing all the information needed for the final calculation.

The grouped mutate deserves special attention as a hybrid operation. When Bryan demonstrates `group_by(country) %>% mutate(le_delta = lifeExp - lag(lifeExp))`, she shows that enrichment can be group-aware: the lag difference is computed within each country, adding a column that captures within-group change. **Grouped mutate is still an "add columns" operation** — it preserves rows while enriching them — but it uses the split-apply-combine machinery under the hood. This is the bridge between the first and second operations.

## Knowing when you're ready to summarize

The summarize step — `group_by() + summarize()` — is the split-apply-combine pattern that Wickham formalized in his 2011 paper: "break up a big problem into manageable pieces, operate on each piece independently, and then put all the pieces back together." In dplyr, `group_by()` is the split, `summarize()` is the apply, and the automatic recombination is the combine. But the pedagogical challenge isn't the mechanics — it's knowing **when to transition from enrichment to summarization**.

The readiness signal is conceptually simple: **you're ready to summarize when each row contains all the variables that the summary function needs**. If you want mean discharge by water year, each row must already have a `water_year` column (the grouping variable) and a `discharge` column (the value to aggregate). If you want discharge normalized by drainage area, each row also needs a `drainage_area` column — which might require a join before you can summarize. Premature summarization — collapsing before rows are self-contained — is one of the most common workflow errors.

Andrew Heiss created animated visualizations specifically to address the conceptual difficulty of summarization: "When I teach about `group_by() |> summarize()`, I end up waving my arms around a lot to explain how `group_by()` puts rows into smaller, invisible datasets behind the scenes." His animations show rows being split into colored groups, then collapsed into single-row summaries — making the split-apply-combine mechanism visual. Bryan's pedagogical sequence teaches summarization in two phases: first simple counting (`tally()`, `n()`), then aggregate functions (`mean()`, `min()`, `max()`). Counting serves as a conceptual bridge because it makes "one row per group" concrete before introducing statistical summaries.

**Summarization changes the unit of observation.** This is the critical conceptual point that distinguishes it from mutate. After `group_by(site, water_year) %>% summarize(mean_q = mean(discharge))`, each row no longer represents a daily measurement — it represents a site-year combination. Understanding this unit-of-observation shift is essential for the third operation, because joins either preserve or further change the grain of the data.

## Joins as the expand operation: denormalization in action

The third operation — expanding via joins — is the least intuitive for students but arguably the most important for real-world analysis. Wickham's tidy data paper frames it directly: "Normalisation is useful for tidying and eliminating inconsistencies. However, there are few data analysis tools that work directly with relational data, so analysis usually also requires denormalisation or the merging the datasets back into one table." **Joins are how tidy data gets reassembled for analysis.**

Joins serve two distinct workflow purposes that students must learn to distinguish:

- **Enrichment joins** add columns from a lookup table without changing the number of rows. Joining airline names to a flights table via carrier code, or joining site metadata (drainage area, land use classification) to a time-series table via site ID. These are conceptually similar to `mutate()` — they add information to each row — but the information comes from another table. R4DS's nycflights13 example is the canonical illustration: `flights` joined with `airlines`, `airports`, `planes`, and `weather` tables, progressively enriching each flight row with context. The dplyr documentation explicitly calls these "mutating joins" — a term that highlights the parallel with `mutate()`.

- **Expansion joins** increase the number of rows by broadcasting coarser-grained data to finer-grained observations. Joining a site table (one row per site) with a monitoring-point table (many points per site) produces one row per monitoring point, each carrying its parent site's attributes. This is the true inverse of summarization: where `summarize()` collapses many-to-one, expansion joins map one-to-many.

The ModernDive textbook teaches joins by explicitly connecting them to the question "wouldn't it be nice to have all this information in a single data frame?" — framing the join as the natural resolution of data spread across tables. The key pedagogical emphasis is on **identification variables that define the unit of observation**: "Key variables are almost always identification variables that uniquely identify the observational units." Students who understand what uniquely identifies a row can predict whether a join will preserve, expand, or potentially corrupt their data's grain.

A particularly powerful teaching pattern is the **summarize-then-rejoin** workflow: compute group-level summaries, then join them back to observation-level data. For example, computing annual mean discharge per site, classifying years as wet/dry/normal, then joining that classification back to daily observations to enable within-class analysis. This round-trip — from observation to summary and back — makes the add-summarize-expand framework feel like a cycle rather than a one-way pipeline.

## Environmental science makes the framework concrete

The three-operation workflow maps onto environmental data analysis with unusual clarity because of two structural features of environmental data: **temporal enrichment is always needed** (calendar dates don't capture hydrological or ecological groupings), and **spatial hierarchies are inherent** (watersheds contain sub-basins contain reaches contain monitoring points).

The USGS "Beyond Basic R — Data Munging" tutorial provides the canonical example. Starting with daily discharge and phosphorus data for the Yahara River, the workflow proceeds through all three operations: add a water year column via `addWaterYear()`, compute cumulative discharge via grouped mutate, summarize annual totals by water year, classify years as wet/dry/normal via `case_when()`, then **join that classification back to observation-level phosphorus data** to analyze nutrient concentrations by hydrologic condition. Every step is motivated by a scientific question, not a coding exercise.

The **fasstr package** (Flow Analysis Summary Statistics Tool for R) from BC Government makes the workflow philosophy architecturally explicit through its function naming convention. All enrichment functions begin with `add_`: `add_date_variables()`, `add_seasons()`, `add_rolling_means()`, `add_daily_volume()`, `add_cumulative_volume()`, `add_basin_area()`. All summarization functions begin with `calc_`: `calc_annual_extremes()`, `calc_longterm_daily_stats()`, `calc_all_annual_stats()`. **The package API itself is a teaching tool** — the `add_*` → `calc_*` naming convention directly mirrors the add-then-summarize workflow. A complete fasstr pipeline chains six `add_*` calls to build up columns, then feeds the enriched data frame to a `calc_*` function.

Spatial hierarchy examples are equally natural. The USGS Stats Service Map example joins current daily discharge values with historical percentile statistics by site number, then adds a category column classifying each site's current condition relative to its historical distribution. A groundwater example from the University of Nebraska demonstrates the full cycle: add month and year columns to well measurements, summarize average depth-to-water by site and year, then join spatially-computed pumping volumes (from nearby wells) back to each observation to enable regression analysis of pumping impacts on water levels.

Three concrete workflows illustrate the pattern's versatility:

- **Exceedance analysis**: Start with daily discharge → add water year, add season → compute exceedance percentiles by site and water year → join site drainage area → normalize to yield
- **Wet/dry classification**: Daily data → add water year → summarize annual totals → classify years → join classification back to daily observations → compare nutrient concentrations across hydrologic conditions
- **Cross-site comparison**: Multiple sites' time series → add water year → summarize 90th percentile by site and year → join site metadata (drainage area, station name, aquifer type) → compare across site characteristics

The USGS `dataRetrieval` package reinforces the framework by providing separate retrieval functions for time-series data (`readNWISdv()`) and site metadata (`readNWISsite()`), naturally requiring joins to combine them. The Virginia Tech Hydroinformatics textbook dedicates an entire chapter to "Joins, Pivots, and USGS dataRetrieval," teaching the three-step pattern in an explicitly hydrological context.

## How the framework relates to split-apply-combine

Wickham's split-apply-combine paper (2011) codified the pattern that `group_by() + summarize()` implements. But the three-operation framework extends split-apply-combine in two directions. First, it positions the "apply" step as potentially either summarization (reducing rows) or grouped mutation (preserving rows) — both instances of split-apply-combine but with different effects on data grain. Second, it adds **joins as the mechanism for crossing between observational units**, which split-apply-combine doesn't address because it operates within a single data frame.

The tidy data paper's four verbs — filter, transform, aggregate, sort — map partially onto the three operations: transform corresponds to "add columns," aggregate corresponds to "summarize," and joins are the multi-table operation the four verbs don't cover. The three-operation framework is thus **a pedagogical synthesis rather than a single published theory**. It draws on the tidy data paper's structure, the split-apply-combine paper's aggregation pattern, dplyr's grammar of mutate/summarize/join, and the tidyverse community's accumulated teaching practices around progressive enrichment. No single Wickham or Bryan paper presents this exact taxonomy, but all the constituent ideas are well-established in their work.

The key insight the three-operation framework adds beyond split-apply-combine is the **expand** operation. Split-apply-combine assumes you start with all the data you need and want to summarize it. The expand step acknowledges that real analysis often requires assembling data from multiple sources or granularities before enrichment and summarization can proceed — and sometimes requires broadcasting summaries back to finer-grained data after summarization.

## Python tools implement the same workflow with different syntax

The three operations map cleanly to both pandas and polars, confirming that the framework is language-independent. Emily Riederer's "Polars' Rgonomic Patterns" (2024) provides the definitive cross-language analysis, arguing that the difference between tools is "switching mindsets — not just syntax." She identifies three pillars of dplyr's ergonomics that polars shares: **consistent** function naming, **composable** return types, and **human-centered** abstraction that lets code read as analytical steps.

The mapping is direct:

| Operation | dplyr | pandas | polars |
|-----------|-------|--------|--------|
| Add columns | `mutate()` | `.assign()` | `.with_columns()` |
| Summarize | `group_by() + summarize()` | `.groupby().agg()` | `.group_by().agg()` |
| Expand via joins | `left_join()` | `.merge()` | `.join()` |
| Window (grouped add) | `group_by() + mutate()` | `.groupby().transform()` | `.with_columns(...over())` |

Tom Augspurger, a pandas core developer, noted in his "Modern Pandas" series that `.assign()` was explicitly **"inspired by dplyr's mutate"** and was added specifically to enable method chaining. Matt Harrison, author of *Effective Pandas*, advocates three rules that enforce the tidy workflow in pandas: avoid index assignment, use `.assign()` for all column creation, and use method chaining for readable "recipe" code. Both emphasize that pandas *can* support this workflow but doesn't enforce it — discipline is required.

Polars is architecturally closer to dplyr. It has no index (eliminating a major source of confusion), returns new DataFrames by default (matching dplyr's immutability), and uses expression-based column references (`pl.col()`) that parallel dplyr's non-standard evaluation. Riederer explicitly recommends polars for R users transitioning to Python because "it may feel more natural and less error-prone." Polars also offers column selectors (`cs.starts_with()`, `cs.integer()`) that mirror dplyr's select helpers, and its `.over()` clause for window functions directly parallels `group_by() + mutate()`.

Riederer's "Column Names as Contracts" adds a deeper layer: standardized naming conventions (prefixes like `ID_` for keys, `AMT_` for summable amounts, `CAT_` for categories) create a "latent contract between data producers and consumers" that enables programmatic operations across all three tools. With consistent naming, `across(starts_with("AMT_"), sum)` in dplyr or `cs.starts_with("AMT_").sum()` in polars can automatically apply the right aggregation — the column structure itself encodes the workflow.

## What goes wrong without the framework

The most common workflow failures cluster into five patterns that the three-operation framework directly prevents:

**Premature summarization** occurs when analysts try to aggregate before rows contain all needed variables. Computing mean discharge by water year before adding the water year column forces awkward workarounds — extracting years manually, subsetting by date ranges, or writing loops over years. The column-accumulation step ensures that grouping variables and analytical variables are in place before summarization begins.

**Row-wise thinking instead of column-wise operations** is the most persistent beginner habit. Students who think "for each row, compute X" instead of "add a column X computed from columns A and B" end up writing loops, `apply()` calls, or incorrect vectorized code. The classic pitfall documented in multiple teaching resources: `mutate(xmean = mean(c(x1, x2, x3)))` computes the grand mean of all values across all rows rather than per-row means, because `mean()` operates on a single vector. The dplyr rowwise vignette frames this as a design philosophy: "dplyr, and R in general, are particularly well suited to performing operations over columns, and performing operations over rows is much harder." Jenny Bryan's exercise ex03 asks pointedly: "Are you SURE you need to iterate over rows?"

**Losing track of the unit of observation** after joins is the subtlest error. When a left join encounters a one-to-many relationship unexpectedly, rows multiply silently, corrupting downstream summaries. R4DS teaches checking keys with `count()` before joining, and modern dplyr throws warnings for many-to-many relationships in equality joins. The pedagogical fix is teaching students to always ask: **"What does one row represent in this data frame?"** before and after every join.

**Leftover grouping** after summarization catches many intermediate users. With multiple grouping variables, `summarize()` drops only the last group level, leaving data still grouped — a behavior Andrew Heiss specifically animated to make visible. Subsequent operations then apply within unexpected groups, producing mysterious results.

**Nested loops replacing vectorized operations** stem from procedural programming habits. Students who learned to program with for-loops instinctively write `for(site in sites) { for(year in years) { ... }}` instead of `group_by(site, year) %>% summarize(...)`. Bryan's famous line captures the antidote: "Of course, someone has to write loops. It doesn't have to be you."

## Conclusion

The add-summarize-expand framework succeeds as a teaching tool because it answers the question students actually have: not "what does `mutate()` do?" but **"what should I do next?"** The answer is almost always one of three things: add a column to bring each row closer to self-containment, summarize because rows now have everything needed for the target calculation, or join to bring in data from another table or granularity. This reduces the cognitive load of data analysis from an open-ended problem space to a structured decision tree.

The framework's deepest insight is that **the unit of observation is the central concept**, not the function call. Every operation either preserves the unit of observation (mutate, enrichment joins), changes it by collapsing (summarize), or changes it by expanding (one-to-many joins). Teaching students to track the unit of observation through a pipeline — "each row is a daily measurement at a site" → "each row is a site-year summary" → "each row is a site-year summary with basin attributes" — provides the conceptual scaffolding that no amount of syntax instruction can replace. The fasstr package's `add_*` / `calc_*` naming convention, the USGS data munging tutorials, and Riederer's cross-language analysis all converge on this same insight: the operations are universal, the syntax is incidental, and the thinking is what matters.
